## 三個關鍵設計問題

### 問題 1：授權（誰能看？怎麼防偷看）

純前端沒有後端可以驗 token。三種策略：

**A. 簽章式 URL（推薦）**
```
?client=ab12cd&t=2026-12-31&sig=hmac(...)
```
- URL 含到期日 + HMAC 簽章
- secret 存在使用者 localStorage（業主拿不到）
- 業主端拆 URL 用 secret 驗 sig
- 缺點：secret 漏了就完蛋；換 secret 等於所有舊連結失效

**B. 一次性 token（穩，需更多基礎建設）**
- 每次產連結就新建一個 random token，存進 `state.shareLinks[]`
- URL 結構：`?share=tok_xxxxxxxx`
- 業主端 → 拿 token 比對 state.shareLinks → 通過才顯示
- 缺點：要每次產連結手動操作、token 管理複雜

**C. 不驗（current）**
- 任何知道 client id 的人都能看
- 缺點：不安全
- 優點：簡單

我推薦 **A 簽章式 URL**，secret 是使用者 GCP Client ID 的 hash 之類（每個使用者唯一），URL 含 expire 時間。

### 問題 2：資料怎麼給業主端？

純前端 + Drive 同步架構下，業主沒有 Drive 權限。三種策略：

**A. 把所有業主資料 inline 進 URL（不可行）**
- URL 太長

**B. 業主端登入自己的 Google 帳號 + 共享 Drive 檔案**
- 接案者把該業主對應的檔案 share 給業主
- 業主端 GIS 登入後讀檔
- 缺點：業主要登入 Google、檔案 share 邏輯複雜

**C. 接案者主動「發布」一份 read-only snapshot 到 public URL**
- 接案者每次想分享前，產一份「該業主資料快照 JSON」push 到 Drive 並 set permission `anyoneWithLink`
- 業主端 fetch 那份 JSON 即可
- 寫回（接受估價單 / 標已付）需要另一個機制（見問題 3）
- 推薦這個

### 問題 3：業主回寫怎麼處理？

業主在介面按「接受案件」「已付款」「留言」要怎麼讓接案者收到？

**A. 業主端把回寫寫到第二份「pending actions」檔案，接案者下次登入 sync**
```
業主操作 → public 寫入 pending-actions-{client_id}.json
         → 內容：[{ type: 'accept-quote', jobId, time }, ...]
接案者登入 → cloudPullPendingActions() → 套用到本地 state → 清掉
```
- 需要對 anyoneWithLink 開放「寫入」（Drive 不建議這麼做）
- 替代：每個業主 share 自己的 Google account 後直接 write

**B. 業主端產 mailto: 連結 → 開使用者 email client → 接案者收 email 手動處理**
- 簡單但體驗差

**C. 用第三方表單服務（Google Forms / Typeform webhook）**
- 業主操作 → submit form → form webhook → 但純前端沒地方接 webhook

**D. 不接受寫回，只能單向看（最務實的 MVP）**
- 業主端純讀
- 寫回機制改成「複製整段內容到剪貼簿」讓業主貼到 LINE / Email 給接案者
- 接案者收到後手動更新狀態

## 推薦的實作路徑（分階段）

### 階段 1：read-only enhanced（v4.0.0）
- 用簽章式 URL（A）
- 寫一份「published snapshot」機制：使用者按「📤 為這業主產分享連結」 → 把該業主資料 export 成 JSON push 到 Drive + 設 anyoneWithLink + 取得 public link → 加 sig 後給業主
- 業主端：拆 URL → fetch JSON → 顯示「合作概況 + 進行中案件 + 待付款項 + 歷史」
- **不接受寫回**：每個 actionable 按鈕變「✉️ 留言給接案者」（複製訊息到剪貼簿 / 開啟 mailto）
- 工程量：大（新 share 機制 + 業主端視圖 + Drive permission API）

### 階段 2：interactive（v4.1.0）
- 業主可以「接受估價單」「標已付」「留言」
- 用「pending actions」檔案機制（Drive 上一份 anyoneWithLink + 業主端 write）
- 接案者下次登入 → pull pending actions → 套用到本地
- 工程量：超大

### 階段 3：refresh on demand（v4.2.0）
- 業主端可以即時看到接案者最新動態
- 加「🔄 重新整理」按鈕、自動 polling
- 工程量：中

## 我的看法

業主入口升級這個方向**價值很高但工程量也很高**。階段 1 read-only enhanced 已經能解決 80% 場景：
- 業主可以隨時看「我們的合作概況」
- 業主可以看待付款項
- 互動回到 LINE / Email 處理（多數使用者也是這樣）

階段 2/3 對「正式接案工作室」很有價值，但對「個人接案略懂程式」的使用者，可能用 LINE / Email 互動就夠了。

**建議**：等實際遇到「業主問你『可以給我看歷史嗎？』」這種需求出現 N 次了再做，不然先解決其他高 ROI 方向（報稅報表、鍵盤快捷鍵）比較實在。
